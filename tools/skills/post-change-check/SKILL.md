# 改动后收口检查

## 说明

在完成代码或文档改动、准备结束当前回合前使用这个 skill。它的目标是减少这种常见失误：

“代码改了，但文档、状态闸门或验证步骤被跳过了。”

开始前先读 [references/closing-checklist.md](references/closing-checklist.md)。

## 工作流

### 1. 先确认改动范围

先列出本次改动，并分组为：

- 代码
- 文档
- 配置
- 测试
- 生成产物

如果仓库有多个模块或业务域，也要指出这次改动命中了哪些范围。

### 2. 再核一遍文档同步

打开仓库治理文档，回答：

- 哪些映射文档本次理论上应该一起变化
- 它们是否已经更新
- 如果没更新，原因是什么

不要静默跳过这一步。

### 3. 再核一遍文档状态闸门

对本次实现依据或被更新到的每一份文档，确认：

- 它是否处于 `已接受`、`已生效` 或 `已落地`
- 如果这次实现实际上依赖了草稿状态文档，要明确报告为流程缺陷

### 4. 再核一遍有没有重复造轮子或边界漂移

对这次改动再确认：

- 是否已经优先复用了现有模块、组件、脚本或工具方法
- 是否新增了没有必要的公共抽象、通用层或包装代码
- 是否让原本清晰的模块边界变得更耦合、更难接手

如果发现“明明能复用却又重写一套”，应明确标记为残余风险或待修正项。

### 5. 执行项目自己的验证方式

优先使用仓库里已经定义好的验证方式：

- `AGENTS.md`
- 本地 skills
- 仓库已有的构建 / 测试脚本

如果仓库已经说明了应使用什么命令，不要临时发明一套新的验证方式。

如果仓库存在 `.doc-sync.json` 与 `scripts/doc_sync_check.py`，默认把 `doc-sync` 检查视为本轮收口的必跑项之一。

如果本次改动命中了公开脚本入口，也要补查对应的 `scripts/*.ps1`、`scripts/*.sh` 和文档命令示例是否仍然一致。

### 6. 输出收口报告

结束前至少说明：

- 改动范围
- 文档同步结果
- 文档状态检查结果
- 复用检查结果，是否存在重复实现或不必要抽象
- `doc-sync` 是否执行以及结果
- 实际执行的构建 / 测试 / lint 命令
- 残余风险或未验证项
