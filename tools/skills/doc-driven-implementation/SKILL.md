# 文档驱动实现

## 说明

当一个仓库把文档当作人类和 AI 的共同操作系统时，使用这个 skill。目标是避免直接基于旧会话、模糊假设或仍处于草稿状态的文档写代码。

如果仓库使用了文档状态闸门，开始前先读 [references/doc-status-model.md](references/doc-status-model.md)。

## 工作流

### 1. 先从仓库文档恢复上下文

除非仓库自己定义了更严格的读取顺序，否则默认按下面顺序读取：

1. `AGENTS.md`
2. `docs/index.md`
3. `docs/evolution/INDEX.md`
4. `docs/governance/document-sync-map.md`
5. 当前任务最相关的 `requirements / design / tasks / upgrade / api`

不要把会话历史当成事实源，仓库文档优先。

### 2. 先判断当前任务类型

在动手前先把任务归类清楚：

- 新需求 / 新能力
- 已有设计的细化
- 基于已接受设计的代码改动
- 只改文档
- 发布 / 升级 / API / SQL 同步

如果是新能力且当前没有设计文档，应先补文档，再推进实现。

### 3. 严格执行文档状态闸门

根据 `references/doc-status-model.md` 判断：

- `草案`、`评审中` 只能用于讨论
- `已接受`、`已生效`、`已落地` 才是有效状态

不要直接根据无效状态文档落代码。

如果仓库使用了不同命名，先做状态映射，再继续。

### 4. 决定这次应先改哪些文档

通过仓库治理文档确认：

- 当前哪些文档是权威入口
- 这次工作应落在 `requirements`、`design`、`RFC` 还是 `tasks`
- 是否还会影响 architecture、API、SQL、upgrade 文档

如果同步矩阵不完整，应先补矩阵，再做大改动。

### 5. 先找现有复用路径，再决定实现落点

在开始实现前，至少回答：

- 当前模块、相邻模块、通用组件、脚本或配置里，是否已经有可直接复用的实现
- 这次应该扩展现有边界，还是确实需要新增抽象
- 如果要新增公共能力，是否已经有明确复用场景支撑

目标不是把每次改动都写成“新架构”，而是优先沿着现有仓库继续演进，减少重复实现和边界漂移。

### 6. 只从有效文档落实现

当任务已经具备实现条件时：

- 明确指出本次实现依据的是哪几份有效文档
- 明确说明本次优先复用了哪些现有实现，或为什么不能复用
- 做最小合理代码改动
- 尽量在同一回合内保持文档与代码同步

如果当前所依赖的设计仍是 `草案` 或 `评审中`，应先停止实现，推动文档进入有效状态。

### 7. 收尾时输出同步说明

结束前至少说明：

- 本次使用了哪些源文档
- 这些文档当前是什么状态
- 本次复用了哪些现有实现，或新增抽象的原因
- 改了哪些代码
- 改了哪些文档
- 还有哪些内容未验证
