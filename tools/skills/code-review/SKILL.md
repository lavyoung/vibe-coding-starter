# 代码评审

## 说明

当目标不是“继续实现”，而是“像 reviewer 一样找问题”时，使用这个 skill。

它和 `post-change-check` 的区别是：

- `post-change-check` 关注收口和流程完整性
- `code-review` 关注缺陷、回归风险、遗漏和不一致

开始前先读 [references/review-checklist.md](references/review-checklist.md)。

## 工作流

### 1. 先确认审查范围

优先审查“本次变更”，不要无边界地泛扫整个仓库。

开始前至少明确：

- 本次改了哪些文件
- 是代码改动、文档改动，还是两者都有
- 当前评审目标是功能正确性、发布安全性，还是流程合规性

### 2. 先确认文档状态和实现依据

在看 diff 前，先回答：

- 本次改动依据的是哪几份文档
- 这些文档当前是否为 `已接受`、`已生效` 或 `已落地`
- 是否存在把草案、评审中内容直接写成现状的风险

如果这是 docs-first 仓库，这一步优先级高于样式和实现细节。

### 3. 再从风险最高的 diff 看

优先检查这些高风险区域：

- 控制流程分支和状态变更
- 新增或修改的接口契约
- 数据结构、字段、DDL、配置变更
- 异步流程、重试、幂等、并发控制
- 上传、删除、支付、权限、回调等敏感路径

不要把大量时间花在低价值样式问题上。

### 4. 再看测试和验证证据

在下结论前，至少确认：

- 本次改动有哪些测试、脚本或人工验证
- 哪些行为变化没有覆盖到
- 评审判断是否依赖未验证假设

### 5. 逐项核查问题类型

至少从这些维度审视本次变更：

- 是否存在明确 bug
- 是否可能引入行为回归
- 是否遗漏异常路径、空态、边界条件
- 是否缺少必要测试或验证
- 是否和已接受 / 已生效文档不一致
- 是否引入了不必要的复杂度或过大改动范围

如果仓库是文档优先模式，还要核对：

- 当前实现依据的文档状态是否有效
- 文档是否把未实现内容写成既定事实

### 6. findings 优先，不写成实现总结

输出时优先给出 findings，而不是先写概览。

每条 finding 应尽量包含：

- 严重程度
- 文件位置
- 问题是什么
- 为什么重要
- 建议修复方向

若没有发现问题，也要明确写出“未发现 findings”，并补充残余风险或未验证项。

### 7. 输出格式固定

默认使用下面的输出顺序：

1. `Findings`
2. `Open Questions / Assumptions`
3. `Residual risk`
4. `Brief summary`

其中：

- `Findings` 必须排在最前面
- `Findings` 按严重程度排序
- 若没有 findings，直接明确说明

## 审查边界

- 不要把评审写成重构提案清单
- 不要把个人风格偏好包装成缺陷
- 不要因为可以继续实现，就跳过问题指出
- 不要在信息不足时假装确定，应明确说明假设和未验证点
