# 代码评审清单

## 1. 先看这次到底改了什么

- 改动是否集中在当前需求范围内
- 是否顺手改到了不相关区域
- 是否新增了没有必要的抽象、配置或复杂度

## 2. 先看文档状态和实现依据

- 本次改动依据的是哪几份文档
- 这些文档状态是否为 `已接受`、`已生效` 或 `已落地`
- 是否把草案、评审中或未实现内容写成既定事实

## 3. 再看会不会出 bug

- 主流程是否存在明显逻辑错误
- 条件分支、状态推进、返回值是否正确
- 新增字段、参数、配置是否全链路对齐
- 上传、删除、提交、回调、支付、权限等敏感操作是否存在遗漏

## 4. 再看会不会回归

- 原有调用方是否会被新改动破坏
- 契约、字段名、状态值、接口含义是否被悄悄改变
- 配置默认值、边界行为、兼容路径是否发生变化

## 5. 再看边界和失败路径

- 空值、空集合、非法输入是否被处理
- 错误态、加载态、空态是否合理
- 异常路径是否会造成脏状态、重复提交或状态丢失

## 6. 再看文档和流程是否一致

- 相关设计文档状态是否为 `已接受`、`已生效` 或 `已落地`
- 代码是否偏离当前有效文档
- 文档是否把尚未实现的能力写成现状
- 若改动命中文档同步矩阵，对应文档是否已更新

## 7. 再看验证是否足够

- 是否有覆盖本次行为变化的测试或验证
- 如果没有测试，是否至少说明了验证方式和缺口
- 评审结论是否依赖未验证假设

## 8. findings 输出要求

- findings 先于总结
- 按严重程度排序
- 引用具体文件位置
- 说明影响，而不是只说“这里不好”
- 没有 findings 时，也要说明残余风险
