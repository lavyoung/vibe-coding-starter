# 项目交接清单（管阶段交接、回合收口和跨人接力时该沉淀什么，准备交接前先查）

## 文档元数据

- 文档类型：project-handoff-guide
- 当前状态：已生效
- 适用阶段：阶段交接、AI 新会话、多人接力、提交前收口
- 最近更新：YYYY-MM-DD

## 1. 这份文档解决什么问题

本文件用于把一次工作结束前最容易散落在会话里的信息，沉淀成稳定交接格式，尽量回答：

- 这轮已经做完了什么
- 哪些事情还没做
- 下一步最应该接着做什么
- 当前风险和未验证项是什么
- 本轮改了哪些代码和文档

## 2. 什么时候应该补这份清单

推荐至少在这些场景补：

- 一次需求或版本阶段暂时收口时
- 准备把任务交给另一个人或另一个 agent 时
- 准备新开会话继续同一件事时
- 提交前需要把本轮边界说清时

如果只是 5 分钟内就能完成的小改动，可以不单独新增一份 handoff 文档，但至少按下面模板把关键信息写在会话结尾、PR 描述或任务记录里。

## 3. 标准交接格式

建议固定用下面这组字段：

### 3.1 已完成

- 本轮已经完成的事项
- 已更新的代码、文档、配置或脚本

### 3.2 未完成

- 明确还没做的事项
- 已知被推迟的工作

### 3.3 下一步

- 下一位接手者最建议先做的 1 到 3 步
- 若存在明确先后顺序，也写清楚

### 3.4 风险与未验证项

- 当前仍不确定的地方
- 还没覆盖的测试、验证或联调场景
- 是否存在依赖草案文档、待确认方案或外部前置条件

### 3.5 文档与代码范围

- 本轮改了哪些代码
- 本轮改了哪些文档
- 哪些文档看过但确认无需更新

### 3.6 验证结果

- 实际跑过哪些命令
- `doc-sync` 是否执行
- `check_all` 是否执行
- 哪些检查没跑，以及原因

若需要稳定交给其他 agent，优先再把以上内容收敛成：

- [../../contracts/handoff-summary.schema.json](../../contracts/handoff-summary.schema.json)
- [../../contracts/examples/handoff-summary.example.json](../../contracts/examples/handoff-summary.example.json)

## 4. 推荐输出模板

```md
## Handoff

### 已完成
- ...

### 未完成
- ...

### 下一步
1. ...
2. ...
3. ...

### 风险与未验证项
- ...

### 文档与代码范围
- 代码：...
- 文档：...
- 无需更新但已检查：...

### 验证结果
- 已运行：...
- 未运行：...
```

## 5. 使用约束

- 不要把“计划做”写成“已完成”
- 不要把草案文档写成当前事实
- 不要只写任务名，尽量写到接手者能直接继续的粒度
- 交接内容应以当前仓库文档和实际改动为准，不以会话记忆为准
