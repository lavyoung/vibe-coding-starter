# UI 文档目录说明（管界面与交互事实源，涉及前端、管理端、控制台页面设计时先查）

## 文档元数据

- 文档类型：ui-directory-guide
- 当前状态：已生效
- 适用阶段：界面设计、交互评审、前端实现、跨角色协作
- 最近更新：YYYY-MM-DD

## 1. 这份目录解决什么问题

`docs/ui/` 用于沉淀“界面层事实”，让人和 AI 在脱离当前会话时，仍能回答这些问题：

- 当前有哪些关键页面、弹窗、配置界面
- 每个页面的字段、按钮、上传限制、状态提示是什么
- 哪些交互模式是通用规则，哪些是单页特例
- 改页面时应该先看哪份文档

## 2. 什么时候启用

这是一个可选目录，不要求所有项目都启用。

适合启用的场景：

- 项目存在 Web 前端、管理端、商家端、运营端、控制台等界面
- 页面字段、上传规则、空态 / 错误态、交互流程需要长期维护
- 需要让产品、设计、前端、后端、AI 共用同一份页面事实源

可不启用的场景：

- 纯后端项目
- 一次性脚本或短期实验项目
- 没有稳定页面和交互资产的小型仓库

## 3. 目录建议

```text
docs/ui/
├── README.md
├── page-map.md
├── interaction-patterns.md
└── screens/
    └── SCREEN_TEMPLATE.md
```

## 4. 分类规则

- `page-map.md`：页面和弹窗总清单，适合作为 UI 层导航入口
- `interaction-patterns.md`：通用交互规则，例如上传、分页、确认弹窗、空态、错误态
- `screens/*.md`：单页面或单场景说明，适合维护字段、按钮、流程、文案和状态

## 5. 使用边界

- 不要把 `docs/ui/` 当成纯截图堆放区
- 不要把外部设计稿链接当成唯一事实源
- 不要把草图、未评审交互写成“已实现现状”
- 页面规则若已成为实现依据，仍需标注状态并经过评审收敛

## 关联代码

- `<frontend/pages/...>`：页面级实现
- `<frontend/components/...>`：复用组件和交互容器
- `<docs/design/...>`：跨后端设计仍以设计文档为准
